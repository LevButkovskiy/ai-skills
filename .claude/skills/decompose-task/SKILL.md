# Decompose Task

Skill для декомпозиции большого проекта или фичи на последовательные атомарные задачи,
каждая из которых выполняется отдельным агентом Claude Code.

## Ключевые принципы из практики

**Один агент — одна зона ответственности.**
Агент не должен знать о существовании других агентов. Его задача — прочитать свой task-файл,
выполнить его, пройти self_check и остановиться.

**Каждый task-файл самодостаточен.**
Агент видит только то, что ему передали. Нельзя писать "смотри analysis.md" без гарантии
что этот файл будет в контексте. Нужный контекст — прямо в task-файле.

**Заглушки лучше блокировок.**
Если модуль A зависит от модуля B которого ещё нет — агент A создаёт заглушку.
Агент B потом заменит её полной реализацией. Никаких ожиданий между задачами.

**self_check обязателен.**
Без проверяемых критериев агент не знает когда остановиться. Каждый task должен иметь
конкретные проверки: команды, SQL-запросы, HTTP-запросы — не субъективные суждения.

## Процесс

### Шаг 1 — Понять масштаб

Выясни из контекста разговора:
- Что за система (новая разработка / перенос / рефакторинг)
- Какой стек (языки, фреймворки, БД)
- Кто ещё работает над проектом (другие разработчики, фронтенд-команда)
- Есть ли уже анализ или документация

Если масштаб непонятен — задай один вопрос. Не несколько.

### Шаг 2 — Определить структуру промтов

Для большинства проектов нужна следующая цепочка промтов:

```
Промт 0: Анализ существующего (если есть)
    ↓ создаёт analysis.md
Промт 1: Генерация CLAUDE.md (соглашения нового проекта)
    ↓ создаёт CLAUDE.md
Промт 2: Генерация задач для агентов
    ↓ создаёт tasks/ папку с task_XX файлами
Промт 3: Шаблон выполнения задачи (используется для каждой задачи)
    ↓ выполняется N раз с разными task-файлами
```

Для простых фич без переноса — начинай сразу с Промта 2.

### Шаг 3 — Определить зоны ответственности

Разбей систему на модули. Типичные зоны для бэкенда:

- Инициализация проекта (скелет, конфиги, docker)
- Схема БД и миграции
- Базовые модули (пользователи, авторизация)
- Бизнес-модули (основная логика)
- Внешние интеграции (боты, SMS, S3, очереди)
- Real-time (WebSocket, события)
- CMS и настройки
- Тесты (unit, интеграционные)
- Финальный аудит

Порядок важен: зависимые модули — позже. БД всегда раньше модулей.

### Шаг 4 — Создать промты

#### Структура каждого промта (XML формат)

```xml
<context>
Состояние проекта на момент запуска этого промта.
Какие файлы уже существуют. Что было сделано до.
Ссылки на файлы которые агент должен прочитать.
</context>

<task>
Одно предложение — что сделать.
</task>

<instructions>
1. Прочитать нужные файлы (всегда первый шаг)
2-N. Конкретные действия
Последний: пройти self_check
</instructions>

<done_criteria>
Задача выполнена только после прохождения всех пунктов <self_check>
из соответствующего task-файла
</done_criteria>
```

#### Структура каждого task-файла (XML формат)

```xml
<goal>Что будет готово после выполнения</goal>

<context>
Выдержки из analysis.md / CLAUDE.md нужные для этой задачи.
Не ссылки — конкретный контент.
</context>

<instructions>
Детальные шаги с примерами кода где нужно.
</instructions>

<dependencies>
- task_XX (причина зависимости)
</dependencies>

<self_check>
1. Конкретная команда или проверка
2. HTTP-запрос и ожидаемый ответ
3. SQL-запрос для проверки данных
</self_check>

<done_criteria>
- Проверяемые пункты без субъективных суждений
</done_criteria>
```

## Паттерны из практики

### Паттерн: Заглушка → Полная реализация

Когда модуль A нужен модулю B, но B ещё не написан:

```
task_03 (Auth) зависит от SmsService:
  → агент создаёт заглушку src/sms/sms.service.ts
  → метод sendOtp только логирует

task_04 (Sms) реализует полную версию:
  → удаляет заглушку, пишет реальный код
```

В промте для task_03 явно указать:
```
2. SmsModule ещё не реализован — создай заглушку с методом sendOtp
   который только логирует. task_04 заменит реализацию позже.
```

### Паттерн: TDD для критичных модулей

Для модулей с бизнес-логикой (ставки, платежи, права доступа):
```
task_XX:
  1. Сначала создай spec-файл с failing тестами
  2. Убедись что тесты RED
  3. Реализуй сервис до GREEN
```

### Паттерн: Conventions файл

Перед генерацией задач создать `conventions.md` (или включить в CLAUDE.md):
- Структура папок
- Правила именования
- ORM и паттерны работы с БД
- Формат API ответов
- Запреты (из слабых сторон старого проекта)

Каждый агент читает этот файл перед стартом.

### Паттерн: Интеграционные задачи

После основных задач — отдельные task-файлы для проверки связей:
```
task_08_integration_check.md:
  - Auth + Users работают вместе
  - Bids → Notifications → WebSocket цепочка
  - Scheduler закрывает аукцион, победитель получает уведомление
```

### Паттерн: Финальный аудит

Последний промт в цепочке — не разработка, а проверка:
- Сборка без ошибок
- Все тесты GREEN
- Swagger полный
- Нет утечек паролей
- Нет заглушек которые забыли заменить

## Правила написания промтов

**Контекст:**
- Всегда указывай какие task-ы уже выполнены
- Перечисляй файлы которые агент должен прочитать явно
- Если модуль-заглушка должна существовать — скажи это

**Инструкции:**
- Первый шаг всегда: прочитать CLAUDE.md и свой task-файл
- Последний шаг всегда: пройти self_check
- Явно указывай что НЕ трогать: "не трогай src/auth/, src/users/"

**Границы задачи:**
- Один агент не должен менять модули других агентов
- Исключение: добавление вызова в уже написанный сервис (указать явно)

**done_criteria:**
- Всегда ссылается на self_check из task-файла
- Никогда не субъективно ("хорошо написано" — плохо, "npm run build без ошибок" — хорошо)

## Что спросить у пользователя перед генерацией

1. Есть ли уже анализ или документация по проекту?
2. Какой стек и какие внешние зависимости?
3. Кто ещё работает над проектом (фронтенд отдельно)?
4. Нужна ли миграция данных или только код?
5. Есть ли conventions / соглашения которые уже приняты?

Не задавай все вопросы сразу — задай самый важный, получи ответ, двигайся дальше.
