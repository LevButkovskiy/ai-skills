# Task Writer

Скилл для структурированного описания задач для трекеров (Jira, Linear, YouTrack и др.).

Перед написанием задачи прочитай эталонный пример: `references/example.md` — там показан правильный стиль и глубина описания.

## Процесс

### 1. Сбор контекста через ask_user_input_v0

Задай уточняющие вопросы одним вызовом инструмента:

```
Вопрос 1: "Для кого пишем задачу?"
Варианты: Для трекера (Jira/Linear/etc) | Для другого разработчика | Для себя (напоминалка)

Вопрос 2: "Насколько подробно?"
Варианты: Кратко (что сделать) | Подробно (причина, что и как делать, результат)
```

Если из контекста уже понятно что-то из этого — не спрашивай повторно.

### 2. При необходимости — уточни детали

Если задача недостаточно описана, задай **один** уточняющий вопрос (не больше), например:
- Какие файлы/модули затронуты?
- Что именно сейчас не работает?
- Какой ожидаемый результат?

### 3. Сформируй задачу

#### Краткий формат
```
## Задача: <название>

**Что сделать:** <одно предложение>
```

#### Подробный формат (использовать по умолчанию для трекеров)
```
## Задача: <название>

**Причина**
<Что сейчас происходит и почему это проблема. 2-4 предложения без технического жаргона.>

**Задача**
<Что конкретно нужно сделать. Коротко и однозначно.>

**Что использовать**
<Библиотеки, инструменты, подходы — если известны.>

**Ожидаемый результат**
- <конкретный измеримый критерий 1>
- <конкретный измеримый критерий 2>
- <ограничения: что не трогаем, если важно>
```

## Принципы

- **Причина** — объясняет *зачем*, не *что*. Пишется так, чтобы понял не только автор.
- **Задача** — одно действие, без деталей реализации.
- **Результат** — конкретные критерии приёмки, не "всё работает".
- Не используй bullet points там, где достаточно одного предложения.
- Технические детали — в "Что использовать", не в "Причине".
- Если контекст уже содержит код или логи — используй их для формулировки, не проси повторно.
